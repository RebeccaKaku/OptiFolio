# OptiFolio 产品总图与执行计划

**状态**：当前最高优先级路线图  
**日期**：2026-06-18  
**适用对象**：项目所有者、Codex（架构/验收）、DeepSeek（小任务实现）

## 1. 一句话定位

OptiFolio 是一个以真实个人资产为中心的投资决策系统：先可信地回答“我有多少钱、赚亏来自哪里、承担了什么风险”，再回答“新钱投哪里、是否需要调仓或替换”，同时把真实持仓变成训练资本市场判断能力的实验室。

它不是：

- 行情终端；
- 自动交易机器人；
- 预测涨跌的 AI 水晶球；
- 家庭全部财富（房产、公司股权等）的估值系统；
- 为了覆盖所有证券而建设的数据平台。

## 2. 项目所有者的真实情况

这些事实是后续设计的约束，不得被 AI 擅自改写：

- 当前纳入范围是个人可支配金融资产；家庭房产、公司股权和家庭实物黄金暂不纳入。
- 主要资产是银行存款、银行理财和公募基金。
- 主要币种是人民币和美元；其他小币种应支持，但可以汇总展示。
- 持有约数万美元的美元计价美债相关银行理财，美元资产约占个人流动金融资产四分之一。
- 产品外壳与实际风险不同：人民币或美元理财/基金可能间接暴露于美债、A 股、美股、黄金等风险。
- 资产录入以手工为主，偶尔只能提供截图；不能假设能够连续每日录入。
- 首次建账可以投入较多时间，之后可能中断数周或数月。
- 用户有金融背景，但只有 Python 入门知识。AI 承担绝大部分工程实现；用户保留范围、假设、风险和最终投资决策权。
- 项目同时承担学习目的：训练资本市场判断，而不是把写代码本身作为主要目标。

## 3. 北极星问题

产品必须按顺序回答六个问题：

1. 我今天确认拥有多少钱？其中多少只是沿用旧净值的估算？
2. 相比上次确认，我赚了还是亏了？来自产品收益、汇率、现金流、费用还是未知差额？
3. 我真正暴露于哪些资产、币种、地区、久期、信用和流动性风险？
4. 某项配置是否相对我的目标区间过多或不足？
5. 新钱投入、调仓或产品替换后，组合会怎样变化？
6. 当时的判断、假设和结果分别是什么，我从中学到了什么？

如果一项开发不能明显改善以上问题之一，默认不做。

## 4. 两条严格分开的轨道

### 4.1 真实资产主干（Production Spine）

用于真实个人资产。要求保守、可解释、可追溯、允许诚实地显示“不知道”。实验模型不得直接修改真实账本或生成无警示的投资建议。

```text
手工录入/截图草稿
        ↓ 人工确认
个人账本 → 可信估值 → 收益归因 → 仓位/风险 → 决策计算器
        ↑
市场与参考数据（FinData，只提供事实，不保存个人持仓）
```

### 4.2 判断实验室（Research Lab）

用于宏观观点、情景分析、回测、优化器、Black-Litterman 和 AI 观点。允许失败，但输出必须标注假设、数据截止日和置信度。

实验结果只有满足以下条件才可“晋升”为主干中的可选输入：

- 有明确金融含义；
- 有样本外或历史情景验证；
- 有压力测试和失效条件；
- 不使用未来信息；
- 用户能读懂建议为何产生；
- 即使模型失败，也不会破坏账本或自动交易。

## 5. 核心架构边界

### 5.1 FinData 的职责

FinData 只负责外部事实：净值、价格、汇率、利率、宏观数据、产品公开资料及其新鲜度和来源。

FinData 不负责：

- 个人账户余额；
- 个人持仓快照；
- 个人现金流；
- 投资目的；
- 用户决策记录。

### 5.2 Personal Book 的职责

新增独立的个人账本边界，建议使用本地 SQLite（例如 `local/portfolio_book.sqlite`）。SQLite 适合可修改的草稿、账户、快照批次、对账和查询；现有追加式 Parquet ledger 不足以承担该职责。

第一版采用“快照 + 事件 + 对账”，不实现完整券商级复式记账：

- **账户**：钱放在哪里；
- **产品**：用户买了什么金融产品；
- **快照批次**：某次确认录入覆盖了哪些账户；
- **持仓快照**：某日数量、市值、成本、币种和数据来源；
- **现金流事件**：申购、赎回、利息、费用、换汇等；
- **对账结果**：两次快照之间可解释和不可解释的变化；
- **导入草稿**：截图/OCR 结果，确认前不得进入正式账本；
- **投资目的/资金桶**：核心、美元用途储备、长期投资、学习实验等；
- **决策记录**：买入理由、基准、情景、失效条件和复盘。

### 5.3 产品与风险暴露必须分离

一笔“美元计价美债银行理财”至少有四层语义：

- 持有形式：银行理财；
- 计价币种：USD；
- 底层风险：美债/利率/可能的信用或结构风险；
- 持有目的：美元用途储备或长期投资。

产品比较、流动性和费用使用产品层；集中度和情景分析使用暴露层。暴露必须携带 `actual/reported/estimated/proxy/unknown` 质量标签。

## 6. 产品界面最终只保留四个主入口

### 我的钱

- 总资产，人民币和原币双口径；
- 账户和产品列表；
- 当日/本周/今年收益（数据不足时诚实降级）；
- 已确认、估算、陈旧和待核对金额；
- 资金流与投资收益分开。

### 我的仓位

- 产品形式、资产类别、币种和底层风险四种视角；
- 集中度、流动性和数据质量；
- 汇率、利率、权益和黄金情景压力测试；
- 核心资产与学习资产分开。

### 决策计算器

- 新钱投入；
- 目标区间缺口；
- 调仓和替换；
- 交易摩擦与“不交易区间”；
- 用户主观观点作为显式、可关闭的输入；
- 输出多套方案和理由，不输出伪精确的唯一答案。

### 产品比较

- 在明确用途下比较存款、货币基金、银行理财、债券基金等；
- 比较到手收益、费用、流动性、久期、信用、汇率和数据质量；
- 不提供脱离用途的统一排行榜。

风险管理贯穿四个入口，不单独堆成指标页面。

## 7. 里程碑与停止条件

### M0：停止打转（文档与范围）

**状态（2026-06-19）**：完成。

**用户得到什么**：所有 AI 看到同一条路线，不再任意扩展数据源或重构。

完成条件：

- 本文档成为最高优先级路线图；
- 新任务必须关联北极星问题；
- 暂停 React、全市场抓取、Qlib、自动交易和新的通用数据源；
- 当前数据审计问题只修复会阻塞“个人账本 → 可信首页”的部分。

### M1：可中断的个人建账

**状态（2026-06-19）**：进行中。DS-001 至 DS-006 已合并并通过测试；先完成 DS-006A/B/C 地基验收，再进入 API 和界面。

**用户得到什么**：可以分多次录入账户、产品和快照，关掉后继续；数周不录也不会破坏系统。

完成条件：

- 可创建账户和产品；
- 可保存草稿和确认快照；
- 每个账户独立显示建账进度；
- 正式记录与 OCR 草稿严格分开；
- 真实数据仅保存在 `local/`；
- 可导出备份并恢复。

### M2：可信的“我的钱”

**用户得到什么**：打开后首先看到个人总资产、币种、账户、数据日期和可信度。

完成条件：

- 人民币与原币双口径；
- 每项资产显示估值日、来源和新鲜度；
- 两次确认快照间区分现金流、市场变化、汇率和未知差额；
- 数据不足时不显示虚假“今日收益”；
- 可以选择个人资产范围，家庭背景不进入数值优化。

### M3：看穿产品外壳

**用户得到什么**：知道基金和理财背后真正承担的股票、债券、黄金、币种、久期和流动性风险。

完成条件：

- 支持手工录入或估计底层暴露；
- 暴露质量和日期可见；
- 美元仓位可以做汇率压力测试；
- 银行理财和基金按产品规则显示流动性；
- 风险提示针对永久损失、未知集中和数据问题，而非对正常小波动大喊大叫。

### M4：新钱计算器与产品比较

**用户得到什么**：输入一笔新资金，得到几种可解释的投向方案；可以比较同用途产品。

第一版不使用均值方差“最优权重”。先采用更稳健的规则：目标区间缺口、流动性约束、币种用途、单产品上限、摩擦和最小交易额。

完成条件：

- 目标使用区间而非单点权重；
- 输出投入后仓位和情景变化；
- 明确区分保留美元与选择美元产品；
- 产品比较展示费用、流动性、风险和数据质量；
- 建议不直接执行交易。

### M5：判断训练闭环

**用户得到什么**：真实持仓成为可复盘的案例，而不是只看盈亏。

完成条件：

- 每个重大决策可记录理由、基准、已定价信息、情景和失效条件；
- 同时显示本币收益、人民币收益和相对基准收益；
- 区分判断、执行、运气和未知因素；
- AI 宏观观点只进入实验区，经过审核后才成为计算器的可选情景。

## 8. DeepSeek 执行规则

DeepSeek 不负责决定架构或任务优先级。每次只领取一个任务包，禁止“顺手重构整个模块”。

每个任务包必须包含：

```text
任务编号与用户价值
允许修改的文件
禁止修改的文件
输入/输出契约
明确非目标
金融语义与关键假设
实现步骤
测试命令
验收标准
失败或疑问的汇报格式
```

每个任务必须同时通过两种验收：

- **工程验收**：测试、迁移、隐私和接口行为；
- **金融验收**：单位、币种、日期、现金流、数据质量和失效场景解释正确。

禁止事项：

- 编造价格、汇率、净值、持仓或产品条款；
- 用默认值掩盖缺失数据；
- 把资金流入当收益；
- 把 OCR 结果直接写入正式账本；
- 把家庭未知资产加入个人净资产；
- 让实验模型自动交易；
- 为了测试通过而固定错误金融数值；
- 在没有迁移计划时更改现有真实本地数据。

## 9. 可交给 DeepSeek 的任务队列

任务按依赖顺序排列。一个任务原则上控制在一次短会话或一个小 PR 内。

### 第一批：个人账本地基

#### DS-001：账本 SQLite 方案与迁移骨架

**用户价值**：让个人数据有一个可恢复、可修改、与市场数据分离的家。  
**依赖**：无。  
**范围**：新增账本数据库连接、schema version 和空迁移；数据库只创建在临时测试目录或 `local/`。  
**非目标**：不迁移旧 YAML，不做 API，不做 UI。  
**验收**：首次启动建库、重复启动幂等、旧/未来版本拒绝策略、临时库测试、隐私扫描通过。

可直接交给 DeepSeek 的完整任务包：`plans/deepseek/DS-001_portfolio_book_sqlite_foundation.md`。

#### DS-002：账户模型与仓储

**用户价值**：可以登记“钱放在哪里”。  
**依赖**：DS-001。  
**最小字段**：`account_id/name/institution/account_type/base_currency/ownership_scope/status/notes`。  
**验收**：创建、读取、修改、停用；名称重复规则明确；不保存账号全号或密码；单元测试覆盖。

#### DS-003：产品模型持久化

**用户价值**：可以登记存款、银行理财和基金，而不是把一切当股票代码。  
**依赖**：DS-001。  
**范围**：复用并扩展 `ProductDefinition` 的持久化映射；支持计价币种、发行方、产品类型、流动性和数据来源。  
**验收**：支持未知字段和缺失条款；未知不得自动填成低风险；往返序列化测试通过。

#### DS-004：快照批次与持仓快照

**用户价值**：建账可以中断并继续，且知道一次录入覆盖了哪些账户。  
**依赖**：DS-002、DS-003。  
**范围**：`draft/confirmed/superseded` 批次状态；持仓记录数量或报告市值；记录 `as_of/source/quality`。  
**验收**：草稿不进入估值；确认后不可静默修改；可用新批次更正；部分账户快照不冒充全组合完整快照。

#### DS-005：现金流事件仓储

**用户价值**：把申购、赎回、转账、利息、费用和换汇与投资收益分开。  
**依赖**：DS-002、DS-003。  
**范围**：扩展 `CashflowEvent` 支持账户间转账配对和外部流入/流出。  
**验收**：内部转账不改变个人净资产；换汇同时保留两种货币金额与费用；重复事件可检测。

#### DS-006：账本备份与恢复

**用户价值**：首次花很久建的账不会因为一次代码更新丢失。  
**依赖**：DS-001。  
**范围**：一致性备份、恢复前校验、schema 版本检查。  
**验收**：临时库完成备份—修改—恢复往返；拒绝覆盖非空库或要求显式确认；备份不进入 Git。

#### DS-006A/B/C：第一批地基验收加固

**状态**：下一期，必须顺序执行，完成后才能开始 DS-007。

**原因**：补齐逐账户快照覆盖、可选份额、现金流财富语义和原子备份恢复。
**任务包**：

- `plans/deepseek/DS-006A_snapshot_coverage_and_optional_quantity.md`
- `plans/deepseek/DS-006B_cashflow_semantics.md`
- `plans/deepseek/DS-006C_backup_integrity.md`

### 第二批：可用的手工建账

#### DS-007：账户与产品 API

**用户价值**：界面可以管理账户和产品。  
**依赖**：DS-002、DS-003。  
**范围**：薄 API，只调用 service；输入校验和错误响应统一。  
**验收**：CRUD API 测试；不在路由中写 SQL 或金融逻辑；私密字段不返回。

#### DS-008：快照草稿/确认 API

**用户价值**：可以分账户保存建账进度，稍后继续。  
**依赖**：DS-004。  
**验收**：创建草稿、追加项目、校验、确认、查询进度；未确认草稿不影响首页；并发/重复确认有明确行为。

#### DS-009：最小建账界面

**用户价值**：无需编辑 YAML 即可录入真实资产。  
**依赖**：DS-007、DS-008。  
**范围**：优先扩展现有本地静态界面，不启动 React 大工程；提供账户列表、产品表单、快照进度和确认页。  
**验收**：浏览器完成一个人民币存款和一个美元理财的草稿—确认流程；刷新后数据仍在；没有网络也能使用。

#### DS-010：截图导入草稿的数据契约

**用户价值**：以后可以用截图减少手工输入，同时保持安全。  
**依赖**：DS-004。  
**范围**：只定义上传、提取候选字段、原图引用、置信度和人工确认状态；先用测试 fixture，不接 OCR 模型。  
**验收**：低置信字段必须提示；截图候选不能进入正式账本；确认过程可逐字段修改。

### 第三批：可信首页与对账

#### DS-011：快照对账引擎

**用户价值**：中断录入后回来，系统能区分已解释和未知变化。  
**依赖**：DS-004、DS-005。  
**输出**：期初、现金流、期末、已解释变化、未知差额、完整度。  
**验收**：资金流入不算收益；缺少现金流时生成 `unclassified_change`；部分账户快照不计算全组合收益。

#### DS-012：估值来源优先级

**用户价值**：每项资产都知道“这个数字从哪来”。  
**依赖**：DS-004，FinData 现有 serving。  
**规则**：人工确认市值、公开净值×份额、沿用旧值等来源明确排序；绝不静默使用假价格。  
**验收**：输出估值日、来源、新鲜度、是否估算；缺失数据返回未知而不是 0 或 1:1 汇率。

#### DS-013：人民币/原币双口径汇总

**用户价值**：同时看到美元资产本身赚了多少和折成人民币后的结果。  
**依赖**：DS-012。  
**验收**：多币种恒等式测试；小币种可汇总但仍计入总额；汇率日期和来源可见。

#### DS-014：收益与汇率归因

**用户价值**：拆出产品本身收益、汇率、交互项、费用和现金流。  
**依赖**：DS-011、DS-013。  
**验收**：归因项可还原总变化；美元正收益但人民币相对收益落后的案例有固定测试；数据不完整时降级为估算或不可归因。

#### DS-015：“我的钱”首页

**用户价值**：项目第一次形成完整闭环。  
**依赖**：DS-011 至 DS-014。  
**范围**：总资产、账户、币种、产品、确认/估算/陈旧/未知金额、最近完整快照。  
**验收**：不得展示技术存储资产列表冒充个人持仓；不得在不满足条件时显示“今日收益”。

### 第四批：风险穿透

#### DS-016：产品暴露快照仓储

记录产品底层资产、币种、地区、久期、信用和黄金等暴露，携带日期、来源和质量标签。

#### DS-017：美元情景分析

对美元兑人民币变化和美元产品本币收益做二维情景表；只展示后果，不预测汇率。

#### DS-018：流动性与永久损失风险

区分正常净值波动、赎回限制、产品信用/结构风险和数据未知；避免把所有波动都标成危险。

#### DS-019：核心/用途储备/学习资金桶

允许同币种、同产品因持有目的不同而采用不同基准和风险规则。

### 第五批：决策工具

#### DS-020：目标区间与仓位缺口

先实现上下限和缺口，不实现“最优权重”。

#### DS-021：新钱投入规则引擎

根据缺口、流动性、币种用途、单产品上限和最小交易额生成多套方案。

#### DS-022：交易摩擦与不交易区间

考虑申赎费、换汇手续费、锁定期和最小交易额；小偏离时建议不交易。

#### DS-023：标准产品比较器

按明确用途比较到手收益、流动性、费用、久期、信用、汇率和数据质量。

### 第六批：判断实验室

#### DS-024：投资决策日志

保存观点、基准、已定价信息、情景、仓位理由、失效条件和复盘日期。

#### DS-025：美元美债理财案例页

把真实案例拆成美元产品收益、人民币换算收益、人民币替代基准和费用；不在测试或文档中保存真实金额。

#### DS-026：AI 宏观观点契约

AI 输出必须包含证据日期、支持/反对证据、情景概率、置信度和失效条件；不得直接输出交易指令。

#### DS-027：研究模型晋升检查清单

将实验模型与真实计算器隔离，只有通过验证和人工审核的版本可注册为可选建议输入。

## 10. 当前明确冻结的工作

在 M2 完成前，不投入主要精力到：

- 新前端框架；
- 全市场每日数据刷新；
- 新国家/新资产类型的抓取器；
- Qlib 或机器学习选股；
- 自动交易；
- 复杂均值方差或 Black-Litterman 界面；
- 房产和公司股权估值；
- 清理与主线无关的所有历史技术债。

安全漏洞、隐私泄露、会造成错误金融结果或阻塞当前里程碑的问题例外。

## 11. 每周工作方式

1. Codex 从当前里程碑挑选一个最小任务，补全任务包。
2. DeepSeek 只实现该任务并提交测试和金融解释。
3. Codex 审查边界、金融语义、测试和与路线图的一致性。
4. 用户只需要确认可见行为和重大金融假设，不需要读懂所有代码。
5. 任务完成后更新本文件的状态，不因为“顺手”增加新功能。

任何时候如果再次感觉打转，先问：

> 这周的变化，是否让“我的钱”更真实，或让一个真实投资决策更清楚？

如果答案是否定的，停止该方向。
