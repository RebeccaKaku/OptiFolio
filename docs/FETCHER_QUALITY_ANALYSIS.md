## BOSC（上海银行）— 并不完美

### 产品发现（`fetch_all_products`）

- POST API（`qryPcFinanceProductZh`）正常返回 293 个产品
- 4 个 GET 端点全部返回 **502 Bad Gateway**（但不影响 POST 的结果）
- 产品元数据完整：prdCode、prdName、prdTemplate、status、isDxFlag 等
- 但 `currNetCycleBeginDate`（成立日期）字段大多为空

### 净值历史抓取（`fetch`）— 🔴 核心问题

**实测：无论产品多老，只返回约 1 个月的近期数据。**

| 产品 | 成立日期 | 请求参数 | 实际返回 |
|------|----------|----------|----------|
| J13163（最老的产品） | 空 | `Month: "36"` | 5 行 (2026-05-06 ~ 2026-05-29) |
| WH2025109A（新产品） | 空 | `Month: "36"` | 31 行 (2026-05-03 ~ 2026-06-02) |
| WH2024101H（最晚成立） | 空 | `Month: "36"` | 31 行 (2026-05-03 ~ 2026-06-02) |

**根因**: `NET_WORTH_URL`（`ebanks.bankofshanghai.com/pweb/FinanceRateChartQuery.do`）
看起来是一个图表展示接口，`Month` 参数可能被服务端忽略，始终只返回近期数据。

这意味着：
- ❌ 无法通过 `sync()` 获取完整历史净值
- ❌ `fetch()` 对回测/优化等场景几乎无用（数据太短）
- ✅ 只能作为**每日增量同步**使用（每天跑一次追加当日数据）

### 其他问题

1. **`verify=False`** — `bosc.py:55` 全局禁用 TLS 证书验证，有安全隐患
2. **非标准接口** — `fetch()` 需要额外 kwargs（`prdTemplate`、`status`、`isDxFlag`），
   无法通过标准 `DataLoader` 路径调用
3. **GET 端点 502** — 4 个 GET API 全部不可用，代码中的 fallback 逻辑形同虚设

### 改进建议

1. **寻找更好的历史数据 API** — 目前的 `FinanceRateChartQuery.do` 是图表接口，
   需要寻找 BOSC 是否有真正的历史净值列表 API（类似 ICBC 的 `getNetValueList`）
2. 如果找不到全量 API，将 BOSC 定位为**仅增量同步服务**，在文档中明确说明
3. 将 `verify=False` 改为可配置参数，默认启用 TLS 验证
4. 移除 4 个永远 502 的 GET 端点调用，减少噪音

## BOC（中国银行）— 数据深度最好

### 产品发现（`fetch_all_products`）

- ✅ 一次 POST 请求返回 2321 个活跃产品
- ✅ 产品代码格式统一（如 `AMHQLXTTUSD01B`）

### 净值历史抓取（`fetch`）— ✅ 优秀

- **单次 GET 请求返回完整历史净值**（无需分页）
- 实测 `AMHQLXTTUSD01B`: 1307 行，2020-08-31 ~ 2026-06-01（~6 年）
- 日期列表 `dateList` + 净值列表 `shareNetWorthList` 同时返回

### 待改进

1. **`sync()` 效率问题** — 遍历 2321 个产品，逐个请求。即使只算成功的情况，
   也会产生大量请求。实测第一个产品 `WFBYZQBO2617` 就无历史数据，
   2321 个产品中很多可能没有净值。
   - 建议：在产品发现阶段过滤掉无净值的产品，或加并发限制 + 批量处理
2. **`fetch_structural_deposits()`** — 用 BeautifulSoup 爬 HTML 页面获取结构性存款代码，
   页面结构变化会导致抓取失败。需要加健壮的错误处理和定期验证
3. **部分产品无数据** — `AMHQLXTTUSD02B` 无历史数据，但未在产品发现阶段标记

## ICBC（工商银行）— 数据可靠但缺少发现能力

### 产品发现 — ❌ 不存在

- **没有公开的产品发现 API**（需要登录才能查询产品列表）
- `sync()` 依赖硬编码默认代码 `["23GS8125", "23GS8123"]` + `processed_dir` 中已有的文件
- 无法自动发现新产品

### 净值历史抓取（`fetch`）— ✅ 稳定

- 分页请求，每页 50 条，按日期倒序排列
- 实测 `23GS8125`: 170 行，2024-01-02 ~ 2026-05-26
- 实测 `23GS8689`: 556 行，2023-12-14 ~ 2026-06-01
- 有总条数 `total` 字段，分页终止条件正确

### 待改进

1. **缺少产品发现** — 这是最大短板。如果 BOSC 或 BOC 能提供产品发现，
   ICBC 需要手动维护产品代码列表
2. **SSL workaround** — 使用 `OP_LEGACY_SERVER_CONNECT` flag 绕过 ICBC 服务器的
   老旧 SSL 配置。这是 ICBC 服务器的问题，但需要在代码中注释说明原因
3. **分页效率** — 历史数据多时需要多次请求（如 23GS8689 需要 12 页），
   但这是 API 限制，实现已正确

## 综合对比

```
数据深度:  BOC >>> ICBC >> BOSC
产品发现:  BOC > BOSC >>> ICBC
接口安全:  BOC = ICBC >> BOSC
标准兼容:  BOC = ICBC >> BOSC
数据可靠性: ICBC > BOC > BOSC
```

| 🔴 P0 | BOSC: 修复 `verify=False` TLS 问题 | 安全隐患 |
